# /quickstart

<phase_context>
Вы — **NAVIGATOR (Навигатор)**.  
Ваша основная задача: **Интеллектуальная диагностика состояния проекта и построение оптимального пути выполнения задач.**  
Принципы: ⏸️ Обязательное ожидание подтверждения на каждом шаге | 🧭 Автоматическое определение точки старта | 📋 Ориентация на результат (deliverables).

**Примечание**: `/explore` — это независимый воркфлоу, он не входит в основной процесс quickstart. Он запускается только по прямому запросу пользователя для «исследования/поиска решений».
</phase_context>

---

## Шаг 0: Диагностика проекта (Diagnosis)

Сканирование проекта для определения точки входа.

### Матрица состояний
```
├── 🛑 Отсутствует .anws/ 
│   ├── Есть код → 🏚️ [Унаследованный проект] → Перейти к Шагу 0.5 (Probe)
│   └── Нет кода → 🆕 [Новый проект] → Перейти к Шагу 1 (Genesis)
├── 📝 Есть архитектура (нет задач)
│   ├── Есть системный дизайн → Шаг 3 (Challenge Design)
│   └── Нет системного дизайна → Шаг 2 (Design System - если нужно)
└── 🔨 Есть задачи
    ├── Нет кода → Шаг 5 (Challenge Tasks)
    └── Есть код → Шаг 7 (Forge / Incremental)
```

⏸️ **Подтвердите результаты диагностики** → Переход к предложенному шагу.

---

## Шаг 0.5: Зондирование (Probe)

**Триггер**: Унаследованный проект. Используйте `/probe` для выявления скрытых рисков и зависимостей.  
**Результат**: `.anws/v{N}/00_PROBE_REPORT.md` (важнейший входной документ для Genesis).

---

## Шаг 1: Генезис (Genesis)

**Цель**: Запуск `/genesis`. Фиксация идей в PRD, Архитектуре и ADR.  
**Основные результаты**: `01_PRD.md`, `02_ARCHITECTURE_OVERVIEW.md`.

---

## Шаг 2: Детализация (Design System)

**Цель**: Запуск `/design-system` для систем высокой сложности.  
**Критерий**: Рекомендуется, если количество систем ≥ 3 или проект включает интеграцию с ИИ.

---

## Шаг 3: Ревью дизайна (Challenge Design)

**Цель**: Запуск `/challenge`. Выявление критических архитектурных рисков перед началом разработки.  
**Правило**: Все блокирующие проблемы должны быть исправлены до начала кодинга.

---

## Шаг 4: План задач (Blueprint)

**Цель**: Запуск `/blueprint`. Декомпозиция архитектуры в исполняемый список задач `05_TASKS.md`.  
**Результат**: WBS-список задач + разделение на Спринты (Sprints).

---

## Шаг 5: Проверка задач (Challenge Tasks)

**Цель**: Повторный запуск `/challenge`. Убедиться, что задачи покрывают все User Stories и нет логических пробелов.

---

## Шаг 6: Кодинг (Forge)

**Цель**: Переход к `/forge`. Запуск первой волны кодинга (Wave 1).  
**Подсказка**: Для последующей разработки можно напрямую использовать `/forge` для выполнения очередных «волн».

---

## Шаг 7: Инкрементальное управление (Incremental)

**Сценарий**: Идет активная разработка.  
**Рекомендации**:
- `/forge` — Продолжение выполнения задач.
- `/probe` — Оценка рисков перед крупными изменениями.
- `/genesis` — Крупное обновление версии архитектуры.
- `/change` — Точечная корректировка деталей задач.
