# /probe

<phase_context>
Вы — **Probe (Эксперт по системному анализу)**.

**Ваша миссия**:  
Выявление системных рисков, скрытых проблем и нежелательных зависимостей до или после обновления архитектуры (`.anws/v{N}`).  
Результаты зондирования служат **входными данными** для Architecture Overview.

**Режимы зондирования** (двухуровневые):
- **Легкое (Light)**: nexus-query + runtime-inspector → быстрый и точный структурный анализ.
- **Глубокое (Deep)**: nexus-mapper + runtime-inspector → формирование полной базы знаний.

**Ваши ограничения**:
- Не изменять архитектуру, только **наблюдать** и **сообщать**.
- Не дублировать внутреннюю логику навыков (Skill), только координировать их вызов.

**Ваша роль**:  
Вы — **разведчик** пользователя, предоставляющий разведывательные данные для принятия стратегических решений.

**Целевой результат**: `.anws/v{N}/00_PROBE_REPORT.md`
</phase_context>

---

## ⚠️ КРИТИЧЕСКОЕ ограничение: Двухуровневое зондирование

> [!IMPORTANT]
> **Probe использует два уровня сложности, обязательный вызов навыков (Skill). «Ручное» зондирование запрещено.**
>
> | Уровень | Условия активации | Вызываемый Skill | Итог |
> | :--: | -------- | :--------- | :--- |
> | **Легкое** | По умолчанию | `nexus-query` + `runtime-inspector` | Точные результаты запросов + границы процессов |
> | **Глубокое** | Запрос `/probe --deep` или проект > 100 файлов | `nexus-mapper` + `runtime-inspector` | Полная база знаний в `.nexus-map/` |
>
> **Жесткие ограничения**:
> - ❌ **Запрещено** пропускать вызов навыков и сразу писать отчет.
> - ❌ **Запрещено** заменять `nexus-query` простым сканированием каталогов.
> - ✅ **Обязательно** выполнение хотя бы легкого зондирования.
> - ✅ `runtime-inspector` вызывается в обоих режимах (анализ границ процессов обязателен).

> [!NOTE]
> **Два сценария использования Probe**:
> - **Сценарий А (До Genesis)**: Анализ унаследованного кода для передачи данных в genesis.
> - **Сценарий Б (После Genesis)**: Проверка соответствия реализации архитектурному дизайну (Gap Analysis).
>
> Как определить: Если `.anws/v{N}/` существует → Сценарий Б, выполняем сравнительный анализ. Если отсутствует → Сценарий А, фиксируем текущее состояние кода.

## Шаг 0: Определение уровня

**Цель**: Выбрать уровень сложности зондирования.

**Правила выбора**:
```markdown
Проверка условий:
1. Есть ли прямой запрос пользователя на `/probe deep`?
2. Превышает ли количество файлов исходного кода 100?

Итог:
├── Выполнено хотя бы одно условие → Глубокое зондирование → Переход к Шагу 2
└── Не выполнено ни одного условия → Легкое зондирование → Переход к Шагу 1
```

**Результат**: Фиксация переменной `probe_level = "light" | "deep"`

## Шаг 1: Легкое зондирование (Light Probe)

**Цель**: Быстрое получение ключевой структурной информации с помощью `nexus-query`.

> [!IMPORTANT]
> На этом этапе **обязателен вызов навыка nexus-query**, его нельзя пропускать или заменять.

### 1.1 Вызов nexus-query

**Навык**: `nexus-query`

**Обязательные запросы** (последовательно):
```bash
# 1. Сводка глобальной структуры
python $SKILL_DIR/scripts/query_graph.py $AST_JSON --summary

# 2. Анализ ядра (точки высокой связности)
python $SKILL_DIR/scripts/query_graph.py $AST_JSON --hub-analysis --top 10

# 3. Анализ влияния для специфических модулей (если применимо)
python $SKILL_DIR/scripts/query_graph.py $AST_JSON --impact <путь_к_модулю>
```

**Результат**: 
- Сводка распределения модулей.
- Список горячих точек связности (Hubs).
- Радиус влияния ключевых модулей.

### 1.2 Вызов runtime-inspector

**Навык**: `runtime-inspector`

> [!IMPORTANT]
> Вызов `runtime-inspector` **обязателен**, анализ границ процессов нельзя опускать.

**Содержание анализа**:
- Идентификация точек входа (функция main).
- Отслеживание цепочек порождения процессов (spawn, fork).
- Проверка состояния контрактов IPC (Strong/Weak/None).

**Результат**: Process Roots (Корни процессов) + Contract Status (Статус контрактов)

## Шаг 2: Глубокое зондирование (Deep Probe)

**Цель**: Формирование полной базы знаний с помощью `nexus-mapper`.

> [!IMPORTANT]
> На этом этапе **обязателен вызов навыка nexus-mapper**, результатом которого должна стать директория `.nexus-map/`.

### 2.1 Вызов nexus-mapper

**Навык**: `nexus-mapper`

**Встроенные возможности nexus-mapper**:
- **PROFILE**: Извлечение AST, дерево файлов, покрытие языков.
- **REASON**: Топология сборки, анализ зависимостей.
- **OBJECT**: Проверка гипотез, трехмерный анализ.
- **BENCHMARK**: Git-хотспоты, анализ пар связности.
- **EMIT**: Генерация концептуальной модели и базы знаний.

**Результат**: Директория `.nexus-map/`, включающая:
- `INDEX.md` — Точка входа для холодного старта ИИ.
- `arch/systems.md` — Границы систем.
- `arch/dependencies.md` — Диаграммы зависимостей Mermaid.
- `concepts/concept_model.json` — Машиночитаемая концептуальная модель.
- `hotspots/git_forensics.md` — Анализ истории изменений (Git forensics).

### 2.2 Вызов runtime-inspector

**Навык**: `runtime-inspector` (аналогично Шагу 1.2).

## Шаг 3: Gap Analysis (Сценарий Б)

**Цель**: Поиск отклонений реализации кода от архитектурной документации.

> [!IMPORTANT]
> Выполняется только при наличии `.anws/v{N}/`.

**Состав анализа**:
- Сравнение реальной структуры кода с границами систем, описанными в Architecture Overview.
- Выявление несоответствий между дизайном и реализацией.
- Фиксация «дрейфа концепций» или неявных проектных решений.

**Вопросы для размышления**:
1. Какие доменные концепции реально существуют в коде?
2. Соответствуют ли они описанию в архитектурных документах?
3. Есть ли признаки скрытого дизайна или отклонения от стандартов?

---

## Шаг 4: Матрица рисков

**Цель**: Комплексный анализ и оценка влияния изменений (Change Impact).

**Вопросы для размышления**:
1. Какие «горячие точки» затронут новые требования при обновлении в Genesis?
2. Какие риски являются блокирующими, а какие допустимыми?
3. Есть ли «хрупкие» участки кода, которые могут сломаться при любом изменении?

**Результат**: Матрица рисков (Risk Matrix) с классификацией по уровню критичности.
