# Мастер концептуального моделирования (Domain Modeler)

> «Если вы не можете что-то ясно описать, вы не сможете это создать». — Эрик Эванс

Этот навык переводит «запросы на уровне ощущений» в четкую доменную модель с помощью **серии уточняющих вопросов**.

---

## 🎯 Миссия и позиционирование

**Суть навыка**: Уточнение размытых требований через взаимодействие с пользователем, построение модели предметной области (сущности, процессы, скрытые компоненты).

**Когда применять**:
- `/genesis` Step 1: Этап уточнения требований.
- Требования содержат двусмысленные термины («синхронизация», «список», «управление»).
- Необходимо сформировать Единый Язык (Ubiquitous Language).

**Когда не применять**:
- Требования уже четко сформулированы и терминология определена.
- Обсуждение касается только технической реализации (без привязки к бизнес-логике).

---

## ⚠️ Ключевой принцип

> [!IMPORTANT]
> **Один сеанс — один вопрос. Не вываливайте на пользователя список всех вопросов сразу.**
>
> **Почему?** Пользователь лучше фокусируется на одном конкретном моменте. Пошаговые уточнения дают более точные ответы и не перегружают собеседника.

---

## 🧭 Интерактивный процесс (Interactive Process)

### Шаг 1: Поиск зон неопределенности

**Цель**: Выявление двусмысленных терминов и недостающей информации.

| Категория | Что проверять |
| :--- | :--- |
| **Размытые сущности** | «Список» — это что? Избранное? Корзина? Список дел? |
| **Размытые глаголы** | «Синхронизация» — односторонняя или двусторонняя? В реальном времени или пачками? |
| **Темная материя** | Пользователь описал только идеальный путь. А обработка ошибок? Авторизация? |
| **Границы** | Кто имеет доступ? Каковы объемы данных и нагрузка? |

---

### Шаг 2: Цикл интерактивных уточнений

**Цель**: Последовательное прояснение каждого «белого пятна».

> [!IMPORTANT]
> **Правила опроса**:
> - Максимум **5 вопросов** за сессию.
> - Каждый вопрос должен быть либо **тестом (выбор варианта)**, либо **кратким ответом (≤5 слов)**.
> - Строго **один вопрос** за один ход ИИ.

#### 2.1 Формат теста (Multiple Choice)
Используйте, если есть четкие варианты сценариев.

#### 2.2 Формат краткого ответа
Используйте для получения уникальных названий или терминов от пользователя.

---

### Шаг 3: Итеративное обновление модели

**Цель**: Мгновенное обновление доменной модели после каждого полученного ответа.

**Что обновляем**:
1. Уточненные сущности → список `entities`.
2. Уточненные действия → список `flows`.
3. Скрытые требования → `missing_components`.
4. Согласованные термины → `glossary`.

---

## 📤 Формат вывода

**Путь**: `.anws/v{N}/concept_model.json`

Пример структуры:
```json
{
  "glossary": {
    "Wishlist": "Список пожеланий пользователя...",
    "Sync": "Двусторонняя синхронизация в реальном времени..."
  },
  "entities": [...],
  "flows": [...],
  "missing_components": [...],
  "clarifications": [
    { "question": "Синхронизация в реальном времени или по расписанию?", "answer": "В реальном времени" }
  ]
}
```

---

## 🛡️ Кодекс Мастера

1. **Никаких предположений**: Никогда не делайте вид, что поняли термин пользователя без подтверждения. Спросите.
2. **Один за раз**: Не перегружайте. Один вопрос — один ответ.
3. **Предлагайте решение**: Дайте рекомендацию с обоснованием, чтобы пользователю было проще выбрать.
4. **Актуальность**: Каждая правка сразу фиксируется в файле, чтобы не потерять контекст при сбое сессии.
5. **Единый язык**: После утверждения термина используйте только его, избегайте синонимов.

## 📋 Критерии завершения

<completion_criteria>
- ✅ Ключевые туманные термины уточнены и внесены в glossary.
- ✅ Отражены основные сущности и их связи.
- ✅ Выявлены скрытые («темные») компоненты системы.
- ✅ Модель сохранена в `.anws/v{N}/concept_model.json`.
- ✅ Пользователь подтвердил правильность понимания терминологии.
</completion_criteria>
