# Заметки Инспектора Соединений (The Wiretapper's Casebook)

> «Код может лгать, но процессы — никогда. Один вызов `.spawn()` говорит больше, чем тысяча строк комментариев». — Пословица старого инспектора.

Ваша задача — **отслеживать линии связи между процессами**.

**Главный закон инспектора**: Если два процесса общаются, но никто не регламентировал их язык, версию или формат — это катастрофа **несоответствия протоколов (Protocol Mismatch)**, которая ждет своего часа.

---

## ⚡ Цели задачи

Выявление **границ выполнения (Runtime Boundaries)** и **коммуникационных контрактов (Communication Contracts)**.

---

## 🧭 Процесс расследования (The Investigation)

### Шаг 1: Поиск точек входа (Entry Points)
Каждая функция `main` может означать отдельный процесс.
- **Цели поиска**:
    - Rust: `fn main()`, `#[tokio::main]`
    - Python: `if __name__ == "__main__":`
    - Node: `package.json` -> `bin`
- **Интуиция инспектора**: Нашли несколько точек входа? Сразу задайте вопрос: «Они работают независимо или управляются родительским процессом?»

### Шаг 2: Отслеживание цепочек порождения (Tracing Spawning)
Если процесс А запускает процесс Б — это «линия родства».
- **Цели поиска**:
    - Rust: `Command::new(...)`, `std::process::Stdio`
    - Python: `subprocess.Popen`, `multiprocessing.Process`
    - Node: `child_process.spawn`
- **Сигнал тревоги (Жизненный цикл)**:
    - ⚠️ «Если родитель умрет, узнает ли об этом ребенок? Есть ли пульс (heartbeat)?» -> **Риск зомби-процессов**.
    - ⚠️ «Если ребенок упадет, перезапустит ли его родитель?» -> **Риск бесшумного отказа**.

### Шаг 3: Прослушивание каналов (Tap the Wire)
Как они «разговаривают»? Где описан протокол?
- **Поиск каналов (Channels)**: Pipe, NamedPipe, Unix Sockets, Shared DB, HTTP/WS.
- **Поиск протоколов (Protocols)**: Handshake, Version, MagicBytes, protobuf, JSON schema.

**Оценка состояния контракта (Contract Status)**:

| Находка | Статус | Рекомендация |
| :--- | :---: | :--- |
| Есть канал + есть `enum Message` или Protobuf | 🟢 **Strong** | Контракт существует, безопасно. |
| Есть канал + есть проверка `Version` или рукопожатие | 🟢 **Strong** | Есть согласование версий. |
| Есть канал + только сырой JSON или строки | 🟡 **Weak** | Нет явного контракта. Опасно при обновлении одной стороны. |
| Есть канал + нет никакого описания протокола | 🔴 **None** | **Черная дыра!** Критический риск. |

## 🛡️ Матрица рисков IPC

| Риск | Признаки | Что делать |
| :--- | :--- | :--- |
| **Protocol Mismatch** | Канал есть, но нет Handshake/Version | **Обязательно** добавить задачу на внедрение рукопожатия. |
| **Zombie Child** | Есть `spawn`, но нет `Kill on Drop` или пульса | Зафиксировать риск утечки ресурсов (процессов). |
| **SPOF (Единая точка отказа)** | Один процесс управляет всем IPC без отказоустойчивости | Предложите логику перезапуска/переподключения. |
| **Named Pipe ACL (Windows)** | Использование именованных каналов без установки прав | 🔴 Критично: по умолчанию доступ может быть у всех (Everyone)! |

## 📤 Итоговый отчет

1. **Process Roots**: Список найденных точек входа (путь, роль).
2. **Spawning Chains**: Иерархия процессов (кто кого породил).
3. **IPC Surfaces**: Найденные поверхности связи (тип, расположение в коде).
4. **Contract Status**: `[Strong / Weak / None]` с обоснованием.
5. **Lifecycle Risks**: Зомби-процессы, бесшумные падения и т.д.
6. **Security Flags**: Настройки безопасности для Windows-каналов.
