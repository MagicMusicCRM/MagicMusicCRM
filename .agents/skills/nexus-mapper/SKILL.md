# nexus-mapper — Протокол ИИ-зондирования проектов

Этот навык обучает ИИ-агента использовать **пятифазный протокол PROBE** для системного исследования любого локального Git-репозитория и создания многоуровневой базы знаний в директории `.nexus-map/`.

---

## Когда вызывать / Когда не вызывать

| Сценарий | Вызывать |
| ------------------------------------------------------------------- | :---: |
| Пользователь предоставил путь к локальному репо и хочет, чтобы ИИ понял его архитектуру | Да |
| Нужно создать `.nexus-map/INDEX.md` для «холодного старта» будущих сессий | Да |
| Пользователь говорит: «Проанализируй проект», «Создай базу знаний», «Изучи этот репозиторий» | Да |
| В среде выполнения нет возможности запускать команды (нет инструмента `run_command`) | Нет |
| На хосте отсутствует Python 3.10+ | Нет |
| Пользователь хочет найти конкретный файл/функцию → используйте `view_file` / `grep_search` | Нет |

---

## Предварительная проверка

О недостающих компонентах нужно явно сообщить пользователю.

| Условие | Способ проверки |
| ----------------- | --------------------------------------- |
| Путь к репозиторию существует | Доступ к `$repo_path` |
| Python 3.10+ | `python --version` >= 3.10 |
| Зависимости скрипта установлены | `python -c "import tree_sitter"` без ошибок |
| Есть доступ к терминалу | Поддержка `run_command` в среде Агента |

## Структура вывода

После завершения в корне репозитория появится папка `.nexus-map/`:

```text
.nexus-map/
├── INDEX.md                    ← Главный вход для ИИ ( < 2000 токенов)
├── arch/
│   ├── systems.md              ← Границы систем + локация кода
│   ├── dependencies.md         ← Граф зависимостей Mermaid + диаграммы
│   └── test_coverage.md        ← Карта тестов и пропуски в покрытии
├── concepts/
│   ├── concept_model.json      ← Машиночитаемая схема (V1)
│   └── domains.md              ← Описание ключевых доменных концепций
├── hotspots/
│   └── git_forensics.md        ← Анализ «горячих точек» Git и связности
└── raw/
    ├── ast_nodes.json          ← Сырые данные анализа Tree-sitter
    ├── git_stats.json          ← Статистика изменений и частоты правок
    └── file_tree.txt           ← Отфильтрованное дерево файлов
```

## Кодекс выполнения

### Правило 1: Отказ от формализма
Основная цель — разрушить «ошибку выжившего». Реальные инженерные факты часто скрыты за названиями папок и историей Git. Догадки должны подтверждаться косвенными уликами из метаданных.

### Правило 2: Реальные пути к коду
При записи в `concept_model.json` необходимо четко разделять статусы: `implemented` (реализовано), `planned` (запланировано) или `inferred` (предполагаемо). Только для реализованных узлов разрешено указывать `code_path`, предварительно проверив его существование.

### Правило 3: Атомарность EMIT
Сначала все файлы записываются в `.nexus-map/.tmp/`. Только при успехе всех операций они переносятся в основной каталог. Это гарантирует отсутствие «битых» баз знаний.

### Правило 4: Роль INDEX.md — Маршрутизатор, а не словарь
`INDEX.md` пишется для **ИИ, который впервые видит этот проект**.
- Строго **меньше 2000 токенов**.
- Обязательный блок «Инструкция по эксплуатации» в конце, предписывающий другим ИИ прочитать конкретные файлы базы знаний перед началом работы.

> «Если в репозитории существует .nexus-map/INDEX.md, обязательно изучи его перед выполнением любых задач по изменению кода».
