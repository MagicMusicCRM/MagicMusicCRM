# Добавление поддержки языков в nexus-mapper

> Этот документ не является обязательным этапом. Если встроенной поддержки языков недостаточно, агенты должны использовать данный справочник для добавления поддержки через аргументы командной строки. Использование JSON-файлов конфигурации рекомендуется только для сложных случаев.

---

## Цель

Текущая модель работы скрипта:
1. Использование встроенных сопоставлений расширений и Tree-sitter запросов.
2. Если встроенной поддержки не хватает, агент добавляет её через командную строку.
3. Если параметров слишком много, используется флаг `--language-config <JSON_FILE>`.

Это означает:
- Нет жесткого требования к наличию файла конфигурации в каждом репозитории.
- Не рекомендуется менять основной скрипт ради разового анализа конкретного репозитория.
- Новый агент может мгновенно добавить поддержку недостающего языка в процессе анализа.

---

## Основной вариант: Добавление через командную строку

### Когда использовать:
- В репозитории появились расширения файлов, не поддерживаемые по умолчанию.
- Нужно добавить от 1 до 3 сопоставлений языков.
- Запрос (query) достаточно короткий для командной строки.

### Шаг 1: Уточнение имени языка
Убедитесь, что имя языка поддерживается в `tree-sitter-language-pack`. Например:
- `.templ` -> `templ`
- `.hbs` -> `handlebars`
- `.rego` -> `rego`

### Шаг 2: Добавление расширения
```bash
python extract_ast.py <repo_path> \
  --add-extension .templ=templ \
  --add-extension .hbs=handlebars
```

### Шаг 3: Добавление запроса (query)
Если нужен анализ структуры (классы/функции), используйте `--add-query`:
```bash
python extract_ast.py <repo_path> \
  --add-extension .templ=templ \
  --add-query templ struct "(component_declaration name: (identifier) @class.name) @class.def"
```

Формат captura должен соответствовать стандартам:
- Классы: `@class.def` / `@class.name`
- Функции: `@func.def` / `@func.name`
- Импорты: `@mod`

## Альтернатива: JSON-конфигурация

### Когда использовать:
- Нужно добавить много языков, и команда становится слишком длинной.
- Запросы (queries) слишком сложные для вставки в терминал.
- Нужно сохранить настройки для повторных анализов.

Пример `language-config.json`:
```json
{
  "extensions": {
    ".templ": "templ"
  },
  "queries": {
    "templ": {
      "struct": "(component_declaration name: (identifier) @class.name) @class.def",
      "imports": ""
    }
  }
}
```

Запуск:
```bash
python extract_ast.py <repo_path> --language-config /path/to/config.json
```

---

## Правила «честного покрытия»

Любой новый язык должен соответствовать уровню детализации в отчетах:
1. **Structural coverage**: Есть парсер и запрос для анализа структур (классов/функций).
2. **Module-only coverage**: Есть парсер, но нет структурного запроса.
3. **Configured-but-unavailable**: Агент запросил поддержку, но в текущей среде парсер не найден.
4. **Unsupported**: Язык не включен в процесс анализа AST.

Запрещено выдавать неподдерживаемые языки за «отсутствующие в репозитории».
